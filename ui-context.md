# Basefolio — UI Context

**Paste this file into every Cursor or Claude Code session when building UI.**
*This is the source of truth for all visual decisions. Do not deviate from these patterns.*

---

## 1. Design Philosophy

Basefolio is a professional tool for operations leads, PMO directors, and startup founders managing real project portfolios. The interface should feel like a well-run operations room, not a consumer app.

**Aesthetic direction: Refined Utilitarian.**
- Information-dense without feeling cluttered
- Every element earns its space
- No decorative chrome, no gradients for the sake of gradients, no illustration unless it's a true empty state
- The product communicates competence. It does not try to be friendly.

**References:** Linear, Retool, Hex. Not Notion, not Monday.com, not Asana.

**What this means in practice:**
- Sidebar is navy (`#1A1A2E`). It is a command surface, not a navigation menu.
- Cards have a 1px border and a subtle shadow. They do not have rounded corners larger than 8px.
- Typography is tight. Line height is compact. This is a dashboard, not a blog.
- Color is used functionally: teal for AI, green/yellow/red for health, blue for primary actions. Not decoratively.
- Whitespace is intentional — used to group related content, not to fill empty pages.

## 2. Color Usage Rules

All colors come from `tailwind.config.ts`. Never use raw hex values in components.

### Background hierarchy
| Layer | Token | Use |
|---|---|---|
| Page | `bg-background` (`#F0F0F0`) | App canvas behind all panels |
| Surface | `bg-surface` (`#FFFFFF`) | Cards, panels, modals |
| Sidebar | `bg-navy` (`#1A1A2E`) | Left nav only |
| Hover | `bg-surface-hover` (`#F8F8FB`) | Row hovers, card hovers |
| Accent bg | `bg-accent-bg` (`#EBF9FD`) | AI callouts, informational banners |

### Text hierarchy
| Use | Token |
|---|---|
| Page titles, card headings | `text-text-primary` |
| Body copy, labels | `text-text-secondary` |
| Timestamps, metadata, helper text | `text-text-muted` |
| Disabled / placeholder | `text-text-disabled` |
| On dark backgrounds (sidebar) | `text-text-inverse` |

### Interactive elements
- Primary button: `bg-primary` + `hover:bg-primary-hover`, `text-text-inverse`
- Secondary button: `bg-surface` + `border border-border` + `text-text-secondary`
- Destructive action: `text-health-red` — text link or outlined button only, never a filled red button
- Links: `text-blue hover:underline`

### Health indicator colors
**Used exclusively for project health status. Do not repurpose these for other UI states.**
```
Green (on track):   bg-health-green-bg  text-health-green  border-health-green/20
Yellow (at risk):   bg-health-yellow-bg text-health-yellow border-health-yellow/20
Red (off track):    bg-health-red-bg    text-health-red    border-health-red/20
```

### AI-generated content
Any content surfaced by Claude (narrative text, risk flags, AI scoring suggestions) must carry a teal accent marker so users always know what's AI vs. human.
- Badge: `bg-accent-bg text-teal text-xs font-medium` with label "AI"
- Left border accent on AI narrative blocks: `border-l-2 border-teal`
- Never label it as "generated by AI" inline — the badge is sufficient

## 3. Typography Rules

Font: **Inter** throughout. Loaded via `next/font/google`.

### Scale in practice

```
Page title (h1):     text-2xl font-semibold text-text-primary    — e.g. "Portfolio Dashboard"
Section heading (h2): text-lg font-semibold text-text-primary    — e.g. "Active Projects"
Card title:          text-base font-semibold text-text-primary   — project name
Body / description:  text-base font-normal text-text-secondary
Label (form, table): text-sm font-medium text-text-secondary
Metadata / timestamp: text-xs text-text-muted
Badge / tag text:    text-xs font-medium
```

### Rules
- **No text larger than `text-3xl` in the application UI.** The marketing site has its own rules.
- **Semibold, not bold, for most headings.** Bold feels heavy at dashboard density.
- **Letter spacing on headings:** the token config includes `-0.02em` on xl/2xl/3xl — don't override this.
- **Line-clamp for multi-line text in cards:** use `line-clamp-2` on project descriptions to enforce predictable card heights.
- **Monospace font for IDs and file names only:** `font-mono text-xs text-text-muted`.

## 4. Spacing & Layout

### Page layout
```
Sidebar width:   240px (fixed, never collapsible in MVP)
Top nav height:  56px
Page padding:    p-6 (24px) on desktop
Content max-width: max-w-7xl, centered within the content area
```

### Component spacing defaults
```
Card padding:        p-5 (20px)
Table row height:    h-12 (48px) with py-3 px-4
Form field gap:      gap-4 between fields
Section gap:         gap-6 between sections within a page
Page section gap:    gap-8 between major page sections
Button padding:      px-4 py-2 (standard) | px-3 py-1.5 (compact)
```

### Grid
- Dashboard uses a **12-column grid** with `gap-6`.
- Stat/KPI row: 4 columns across on desktop (`col-span-3` each), 2 on tablet.
- Project list: full width, single column.
- Project detail: 8-column main content + 4-column sidebar.

### Pattern 3: Health Indicators

Three variants: **badge** (cards, tables), **dot** (compact rows), **full pill** (detail pages).

```tsx
// Health badge — use on cards and in table cells
const healthConfig = {
  green:  { label: 'On Track',  bg: 'bg-health-green-bg',  text: 'text-health-green',  border: 'border-health-green/20' },
  yellow: { label: 'At Risk',   bg: 'bg-health-yellow-bg', text: 'text-health-yellow', border: 'border-health-yellow/20' },
  red:    { label: 'Off Track', bg: 'bg-health-red-bg',    text: 'text-health-red',    border: 'border-health-red/20' },
}

// Badge
<span className={`inline-flex items-center gap-1.5 px-2 py-0.5 rounded text-xs font-medium border
                  ${config.bg} ${config.text} ${config.border}`}>
  <span className={`w-1.5 h-1.5 rounded-full bg-current`} />
  {config.label}
</span>

// Dot only — for compact list rows
<span className={`w-2.5 h-2.5 rounded-full shrink-0 ${dotColor}`} />

// Full pill — for project detail header
<span className={`inline-flex items-center gap-2 px-3 py-1 rounded-full text-sm font-medium border
                  ${config.bg} ${config.text} ${config.border}`}>
  <span className="w-2 h-2 rounded-full bg-current" />
  {config.label}
</span>

// RULE: Never use color alone to convey health. Always include the text label
// in badge and pill forms. Dot-only is acceptable only in dense list rows
// where the row context makes status clear.
```

### Pattern 5: Empty States

Every list view needs an empty state. This is where most AI-generated UIs look amateurish — either missing entirely or generating generic "No data found" text.

**Basefolio empty state formula:**
1. Icon (48px, `text-text-disabled`)
2. Heading — what's missing, plainly stated
3. Body — one sentence explaining what this thing is for
4. Single primary action (the most logical next step)

```tsx
// Empty state component
<div className="flex flex-col items-center justify-center py-16 px-8 text-center">
  <div className="w-12 h-12 rounded-lg bg-gray-light flex items-center justify-center mb-4">
    <FolderOpen className="w-6 h-6 text-text-disabled" />
  </div>
  <h3 className="text-base font-semibold text-text-primary mb-1">
    No active projects
  </h3>
  <p className="text-sm text-text-muted mb-6 max-w-xs">
    Projects you're tracking will appear here. Add your first project or 
    import from your last Status Draft.
  </p>
  <div className="flex items-center gap-3">
    <Button variant="primary">Add Project</Button>
    <Button variant="secondary">Import from Status Draft</Button>
  </div>
</div>

// Contextual empty state copy by view:
// Dashboard:   "Your portfolio is empty" / "Add your first project to start tracking health and progress."
// Intake queue: "No pending requests" / "Requests submitted through your intake form will appear here."
// Risk register: "No risks logged" / "Risks you log or that AI surfaces from status updates will appear here."
// Reports:    "No reports generated" / "Run Status Draft to generate your first portfolio report in minutes."
```

//   toast confirmation: "Status update saved."

## 6. Microcopy & Voice

AI-generated UI copy is where "generic" is most visible. These rules produce copy that sounds like Basefolio, not ChatGPT.

### Rules

**Write actions, not features.**
- Not: "Project Management" → Yes: "Add a project"
- Not: "Reporting Module" → Yes: "Generate your status report"

**Empty states explain value, not absence.**
- Not: "No projects found" → Yes: "Projects you're tracking will appear here"
- Not: "No data available" → Yes: "Run Status Draft to see your portfolio in minutes"

**Errors tell the user what to do.**
- Not: "File upload failed" → Yes: "We couldn't read that file. Export it as .xlsx or .csv and try again."
- Not: "Invalid input" → Yes: "Status is required. Choose Green, Yellow, or Red."

**Confirmations are specific, not cheerful.**
- Not: "Great! Your project was saved." → Yes: "Project saved."
- Not: "You're all set!" → Yes: "Status update added for week of April 14."

**Never say "please" in UI copy.** It's patronizing, not polite.

**AI content is always attributed, never hidden.**
- AI-generated narrative: always tagged with the teal AI badge
- AI risk flags: "AI surfaced this risk from your status updates" — one line, small type
- AI scoring suggestions: "Suggested by AI · You can override this"

### Button labels

| Action | Label |
|---|---|
| Save a new record | "Add [thing]" |
| Save edits | "Save Changes" |
| Generate AI content | "Generate" or "Draft with AI" |
| Confirm a destructive action | "Delete [thing]" — specific, never just "Delete" |
| Cancel | "Cancel" — never "Nevermind", "Go back", "Dismiss" |
| Close a modal | "Done" |
| Navigate to a full detail view | "View [thing]" |

### Loading states

- File upload: "Parsing your file..." → "Identifying projects..." → "Computing metrics..."
- Status Draft generation: "Generating your status report..." (streaming renders immediately after)
- Any action under 500ms: no loading state. Optimistic UI only.
- Anything over 500ms: subtle spinner inline with the button, not a full-page overlay.

## 7. Do Not Do List

Generated code should never include these patterns.

- **No gradients in the app UI.** Marketing site only.
- **No shadow-xl or larger on cards.** shadow or shadow-md maximum.
- **No border-radius larger than rounded-lg (12px) on cards.** rounded-xl is for modals/feature callouts only.
- **No full-page loading spinners.** Skeleton loaders or optimistic UI only.
- **No toast stacks.** One toast at a time, bottom-right, 4s auto-dismiss.
- **No red filled buttons.** Destructive actions are text or outlined.
- **No raw health colors used for non-health purposes.** green/yellow/red are reserved for project health indicators only.
- **No emojis anywhere in the product UI.**
- **No placeholder avatar images.** Use initials in a colored circle (`bg-blue text-white`).
- **No `text-black`.** Use `text-text-primary` (`#1A1A1A`).
- **No inline styles.** All styling through Tailwind tokens.
